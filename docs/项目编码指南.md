# 项目发布包上传及自动化部署工具实施计划

## 目标说明
设计并实现一个集“发布包上传”、“自动化部署”、“运行日志查看”于一体的工具。该工具需支持：
- 中心化中转与分发：开发人员上传至中转服务器 (1.50 Windows)，再由中转服务器分发至目标 Linux 测试服务器 (如华为云)。
- 级联备份机制：中转服务器保留 7 天备份，目标服务器保留近期备份（3 天）。
- 跨平台部署：支持从 Windows 控制台远程操作 Linux (华为云等) 环境。
- 运行日志：支持通过中转服务器透传或直接连接目标机实时查看日志。

## 方案设计

### 1. 系统架构
采用 **中心控制台 (1.50) + 多目标节点 (Linux Servers)** 的模式。

- **Web 控制台 (部署于 1.50 Windows)**: 
  - 核心存储：接收开发人员上传的资源包并存放在指定卷。
  - 安全隔离：作为唯一具备 SSH 访问目标机 B 权限的跳转中心，开发人员无需也不具备直连 B 节点的权限。
  - 分发逻辑：负责将资源包通过 SCP 或 SSH 流分片分发至各目标测试服务器。
- **目标服务器 (Linux/华为云)**:
  - 真正对外提供业务服务的环境。
  - 执行具体的 Linux 命令进行发布（备份、解压、重启）。
- **自动化执行层**:
  - `Node.js` 控制台驱动跨平台的 SSH 指令执行。
- **日志采集器**:
  - 通过远程文件流（Tail -f 风格）实时读取目标服务器上的日志文件。

### 2. 技术栈
- **前端控制台**: Next.js (React) + Element Plus (Tailwind-compatible styles) + Lucide Icons.
- **后端服务**: Node.js (Next.js API Routes).
- **远程操作**: `ssh2` (Node.js 库) 用于与 Windows Server 通讯。
- **数据库**: MySQL (公司内部标准数据库，用于项目配置、服务器信息、发布记录)。
- **部署脚本**: Shell Script (在 Linux 目标机器 B 上执行备份与重启)。

### [Component] 配置与数据存储
#### [IMPLEMENTED] `Database Schema (MySQL)`
- `projects`: 存储项目名、Git/模块信息、基准路径 (`base_path`)。
- `modules`: 存储模块名称、类型 (jar/zip)、远程路径、启动/停止/重启命令。

- `user_project_permissions`: 用户与项目权限关联表（RBAC 权限控制）。

### 2. 技术实现要点 (Technical Key Points)
- **跨平台 SSH 协同**: 
  - 控制台 (Windows 1.50) 通过 `ssh2` 驱动 Linux (B 节点) 执行 `tar`, `mv`, `systemctl` 等命令。
- **级联备份策略**: 
  - **1.50 端**: 存档最近 **7 天** 的历史上传包，超期自动清理。
  - **B 节点端**: 维护一个 `backups/` 目录，通过脚本保留最近 **3** 天的包，超期自动删除。
- **大包分发逻辑**: 
  - 传输链路：开发机 -> 1.50 (HTTP POST) -> B 节点 (SCP/Stream)。
  - 利用 Linux 环境的 `rsync`（若有）或 `scp` 保证分发的高可靠性。
- **非侵入式日志查看**: 
  - 通过 `ssh` 远程执行 `tail -f` 捕获 stdout，实时推送到 Web UI，无需在 B 节点安装额外 Agent。
- **发布审计与回滚**: 
  - 记录每次发布的版本。若 B 节点服务异常，可一键触发布脚本从 B 节点本地备份中快速恢复最近版本。

## 关键技术实现细节 (Key Implementation Details)

### 1. 日志类型分类系统

**目的**: 区分部署、重启、停止三种不同类型的服务器操作，便于审计和统计分析。

**数据库迁移 SQL**:
```sql
-- 添加 log_type 字段
ALTER TABLE deploy_logs 
ADD COLUMN log_type VARCHAR(50) DEFAULT 'deploy' 
AFTER status;

-- 更新已有记录
UPDATE deploy_logs 
SET log_type = 'deploy' 
WHERE log_type IS NULL;
```

**API 实现要点**:
- **部署 API** (`/api/deploy/route.ts`): 
  - 成功时插入: `log_type = 'deploy'`, `status = 'success'`, 包含 `version`
  - 失败时插入: `log_type = 'deploy'`, `status = 'failed'`, 包含 `log_output` 错误信息
  
- **重启 API** (`/api/deploy/restart/route.ts`): 
  - 成功时插入: `log_type = 'restart'`, 记录完整执行命令
  - 失败时插入: `log_type = 'restart'`, `status = 'failed'`, 记录异常信息
  - 包含 `start_time` 和 `end_time` 时间戳
  
- **停止 API** (`/api/deploy/stop/route.ts`): 
  - 成功时插入: `log_type = 'stop'`
  - 失败时插入: `log_type = 'stop'`, `status = 'failed'`
  
- **查询 API** (`/api/deploy-logs/route.ts`): 
  - 支持 `logType` 查询参数进行筛选
  - 分页计数查询同步应用 `log_type` 过滤条件

**前端展示**:
- **首页 Activity Feed** (`src/app/(dashboard)/page.tsx`):
  - 使用颜色编码标签区分类型
  - 发包 (deploy): 蓝色/靛青 `#e0e7ff`
  - 重启 (restart): 橙色 `#ffedd5`
  - 停止 (stop): 玫红 `#ffe4e6`
  
- **历史页** (`src/app/(dashboard)/history/page.tsx`):
  - 新增"操作类型"表格列
  - 新增"操作类型"筛选下拉框
  - 详情弹窗显示操作类型标签

### 2. 手动服务控制优化

**问题**: 初始实现的重启/停止按钮没有在正确的工作目录下执行命令，导致相对路径命令失败。

**解决方案**: 添加目录上下文切换逻辑
```typescript
// 计算有效远程路径
const basePath = module.project_base_path || '';
const rawRemotePath = config?.remote_path || module.remote_path;
const effectiveRemotePath = path.join(basePath, rawRemotePath || '');

// 在命令前添加 cd 切换目录
const fullRestartCmd = `cd "${effectiveRemotePath}" && ${restartCmd}`;
await ssh.exec(fullRestartCmd);
```

**错误处理增强**:
- 所有手动操作均记录到 `deploy_logs` 表
- 失败操作同样写入日志，包含异常堆栈信息
- 使用嵌套 try-catch 确保即使日志记录失败也不影响错误返回

### 3. 环境特定配置覆盖机制

**设计思路**: 同一模块在不同环境（开发/测试/生产）可能需要不同的启动参数或路径。

**实现方式**:
- `module_env_configs` 表存储环境特定配置
- API 查询时优先级: 环境特定配置 > 模块默认配置
```typescript
const effectiveRestartCommand = config?.restart_command || module.restart_command;
const effectiveStartCommand = config?.start_command || module.start_command;
const rawRemotePath = config?.remote_path || module.remote_path;
```

**支持字段**:
- `restart_command`: 重启命令
- `start_command`: 启动命令
- `stop_command`: 停止命令
- `remote_path`: 远程路径

### 4. 首页统计准确性修复

**问题**: "今日发布数量" 使用 UTC 时间计算，与实际本地时间不符。

**修复方案**: 使用服务器本地时间计算当日范围
```typescript
const now = new Date();
const startOfToday = new Date(now.getFullYear(), now.getMonth(), now.getDate());

const [todayRows] = await pool.query(
    'SELECT COUNT(*) as count FROM deploy_logs WHERE start_time >= ?',
    [startOfToday]
);
```

## 潜在风险与设计规备 (Risk Analysis)

| 风险点 | 影响 | 规避措施 |
| :--- | :--- | :--- |
| **部署原子性** | 替换包途中网络中断导致服务挂死 | 采用“安全替换”策略：先通过 `mv` 重命名备份，再传输新包，随时可快速改名恢复。 |
| **凭证泄露** | 数据库泄露导致目标机失控 | 对 SSH 密码/密钥进行 **AES 对称加密存储**，后端动态解密使用。 |
| **磁盘撑爆** | 频繁发布导致空间不足 | 强制执行 **1.50(7天) / B节点(3天)** 自动清理规则。 |
| **并发冲突** | 两人同时发布导致状态混乱 | 后端逻辑中引入 **发布互斥锁**，针对同一模块的任务需排队。 |
| **脚本换行符** | Windows 编辑的脚本在 Linux 报错 | 下发脚本前强制进行二进制流转换，确保为 `LF` 换行。 |

## 用户评审要求
1. **网络白名单**: 1.50 (Windows) 到 B 节点 (Linux) 的端口 22 (SSH) 是否已连通？
2. **目标机权限**: 控制台连接 B 节点使用的 SSH 账号是否具备操作目标路径及重启服务的 `sudo` 权限？
3. **备份天数**: 目标服务器 B 上的备份保留 **3** 天。
